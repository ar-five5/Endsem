## Slide 24: Software Products

24
Software Engineering
Software engineers are concerned with developing software products (software that can be sold to a customer).
Kinds of Software Products:
  - Generic Product: These are stand-alone systems that are produced by a development organization and sold on the open market to any customer. Example: MS Word
  - Customized (or bespoke) Product: These are systems that are commissioned by and developed for a particular customer. Example: Custom ERP, ATC software
The important distinction between the two is that in Generic software, the software specification is controlled by the organization that develops the software, whereas in the second case, the specification of the software is controlled by the client or the customer buying the product (how can this be a challenge?)

## Slide 26: Software Engineering, more formally

26
Software Engineering
"Software engineering is an engineering discipline that is concerned with all aspects of software production from the early stages of system specification through to maintaining the system after it has gone into use" (Ian Sommerville, 10th Edition)
Two key aspects here:
  - Engineering discipline: Using appropriate theory and methods to solve problems bearing in mind the constraints and costs (Engineering is about getting results of the required quality within schedule and budget)
  - All aspects of software production: Software engineering is not just concerned with the technical processes of software development (design and code), but also activities such as software project management and the development of tools and methods in various phases of software production

## Slide 27: Software Engineering v. Physical Engineering

27
Software Engineering
Physical systems are often continuous in the sense that small changes in the specification lead to small changes in the product. This is not true with software
Small changes in the specification of software may lead to considerable changes in the software itself
In a similar way, small errors in software may have considerable effects. The Mariner space rocket to Venus, for example, got lost because of a typing error in a FORTRAN program (Read here)
In 1998, the Mars Climate Orbiter got lost, because one development team used English units such as inches and feet, while another team used metric units (Read case study: Lost in Translation – no, not the 2003 Sofia Coppala film)

## Slide 34: Implications of Changing Nature of Software

Some key implications:

## Slide 39: Software Engineering Ethics

Areas where acceptable behavior might not be bound by laws, but instead by the tenuous notion of professional responsibility
  - Confidentiality: Respect confidentiality of employer and client whether or not a formal agreement has been signed
  - Competence: Don't misrepresent competence, don't accept work outside your level of competence
  - IPR: IP of employers and clients must be protected
  - Computer misuse: Don't use your skills to misuse other's computers/ systems
Organizations like ACM and IEEE have cooperated to produce a joint code of ethics and professional practice

## Slide 41: Software Process

41
Software Engineering
The systematic approach that is used in software engineering is sometimes called a software process. Four fundamental activities are common to all software processes:
  - Software specification, where customers and engineers define the software that is to be produced and the constraints on its operation
  - Software development, where the software is designed and programmed
  - Software validation, where the software is checked to ensure that it is what the customer requires
  - Software evolution, where the software is modified to reflect changing customer and market requirements
Ian Sommerville, 10th Ed.

## Slide 47: Software Engineering Fundamentals

47
Software Engineering
Some fundamental principles apply to all types of Software system, irrespective of the development techniques used:
  - Systems should be developed using a managed and understood development process. Of course, different types of processes are used for different types of software

  - When appropriate, you should reuse software that has already been developed rather than write new software

## Slide 48: Case study 1 – Insulin pump control system

This is an Embedded Control System – Software Controls a hardware device and is embedded (stored in permanent memory) inside the device
Important factors in general embedded systems:
  - Size: Cannot to too large, since memory limited
  - Responsiveness: Since it is interacting with a user via a physical device, must be fast / real time (Ex: Command to display turn right on display should not happen after the lane/ road has passed by)
  - Power Management: Many a device run on battery, and in that case, power should be managed efficiently (Ex: In a sensor node that sends indoor temp. readings, better send data less frequently to save radio transmissions and battery)
